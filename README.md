# Cassava Leaf Disease Classification — ML Engineering Assessment

End-to-end **computer vision** prototype for cassava leaf images: **data preparation**, **two-model training**, **evaluation & error analysis**, **inference (script + API)**, optional **Docker** for the API, and a **Streamlit dashboard** to review artifacts and rerun steps.

Python **3.10+** recommended (Dockerfile uses **3.11**).

**Full written report (interview / submission):** [`ASSESSMENT_TECHNICAL_REPORT.md`](./ASSESSMENT_TECHNICAL_REPORT.md) — paste your GitHub link in section 8, then open in Word or PDF if needed.

---

## What this solves

- **Task:** Multi-class image classification (`train.csv`: `image_id`, integer `label`; 5 disease/healthy classes).
- **Dataset:** Kaggle Cassava Leaf Disease Classification (training images under `data/raw/train_images/` after unzipping).
- **Split:** Stratified train / validation / test CSVs produced in Deliverable 1 (`data/processed/splits/`).

---

## Models used — and why

| Model | Role | Why |
|--------|------|-----|
| **Baseline CNN** (`BaselineCNN`) | Required “simple baseline” | Small CNN trained from scratch; shows you can optimize without pretrained weights and gives a baseline for comparison. Fast on CPU relative to large architectures. |
| **ResNet-18 + transfer learning** (`transfer`) | Required “stronger” model | ImageNet pretrained features fit limited data and compute better than training a deep backbone from scratch. **ResNet-18** is a standard, defensible balance of accuracy vs size/speed versus very large nets. |

**Transfer learning rationale:** Strong class imbalance and real-world variability; pretrained low/mid-level features help under typical laptop timelines. Only **layer4 + classifier** are tuned to stay compute-aware.

**Metrics:** Validation and test metrics emphasize **accuracy** plus **macro-F1** so minority classes are not masked by majority-class accuracy.

## Environment setup

### 1) Clone / open repo and create a virtual environment (recommended)

```bash
cd Assessment
python -m venv .venv
# Windows:
.venv\Scripts\activate
# Linux/macOS:
# source .venv/bin/activate
```

### 2) Install Python dependencies

```bash
pip install -r requirements.txt
```

Place Kaggle assets under **`data/raw/`** (see Deliverable 1 assumptions in scripts): `train.csv`, `train_images/`, `label_num_to_disease_map.json`, etc.

### 3) Optional — AMD Radeon GPU on Windows (DirectML — not NVIDIA CUDA)

PyTorch’s default `pip install torch` is **CPU-only** on Windows for many setups; **CUDA is for NVIDIA only**.  
For **AMD discrete GPUs**, use **torch-directml** (matches a specific PyTorch stack; may replace previously installed torch/torchvision):

```bash
pip install torch-directml
python -c "import torch_directml; print(torch_directml.device_name(0))"
```

Official guide:  
[GPU-accelerated ML training with PyTorch and DirectML (Windows)](https://learn.microsoft.com/en-us/windows/ai/directml/gpu-pytorch-windows)

Training script supports:

```bash
python src/train/train_models.py --model both --fast --device directml
# or automatic: CUDA first, then DirectML, then CPU:
python src/train/train_models.py --model both --fast --device auto
```

## How to run (CLI)

Assume project root contains `src/` and data paths from `configs` / defaults.

### Deliverable 1 — Data preparation

```bash
python src/data/prepare_dataset.py --check-noise
```

Outputs summaries under `artifacts/reports/` and `data/processed/`.

### Deliverable 2 — Training

**Faster preset (recommended for demos / limited time):**

```bash
python src/train/train_models.py --model both --fast --device auto
```

**CPU-only:**

```bash
python src/train/train_models.py --model both --fast --force-cpu
```

**Higher quality but slow on CPU (long run):**

```bash
python src/train/train_models.py --model both --epochs 8 --batch-size 32 --tune --tune-epochs 2 --max-tune-trials 3 --force-cpu
```

Checkpoints:

- `artifacts/training/models/baseline_best.pt`
- `artifacts/training/models/transfer_best.pt`

### Deliverable 3 — Evaluation

```bash
python src/eval/evaluate_models.py --force-cpu
```

Optional smaller test subset during debugging:

```bash
python src/eval/evaluate_models.py --force-cpu --max-test-samples 500
```

### Deliverable 4 — Inference CLI

Uses `transfer_best.pt` unless you override flags:

```bash
python src/infer/predict_image.py --image path/to/image.jpg --checkpoint artifacts/training/models/transfer_best.pt --label-map data/raw/label_num_to_disease_map.json
```

Prediction JSON default: `artifacts/inference/last_prediction.json`.

### Deliverable 4 — FastAPI (local)

```bash
uvicorn src.infer.api:app --host 0.0.0.0 --port 8000
```

`POST /predict` with multipart file field `file` — image upload.

Example:

```bash
curl -X POST "http://127.0.0.1:8000/predict" -F "file=@sample.jpg"
```

Configure paths in **`configs/inference_config.json`** (checkpoint, label json, CPU flag for API startup).

## Docker — inference API only

Training is intentionally **not** in the container image (datasets are large); the image exposes the **inference FastAPI** app matching the assessment deployability criterion.

Ensure `requirements.txt`, `configs/inference_config.json`, and checkpoints are present relative to context (Docker build copies repo root).

From project root:

```bash
docker build -t cassava-inference .
docker run --rm -p 8000:8000 cassava-inference
```

Browse `GET /health`; send images to `POST /predict`.

**Note:** The default PyTorch wheels in Docker are CPU-based for portability. For cloud GPU deployments you would swap the base image or install CUDA-enabled PyTorch in a CUDA base image — not required for this prototype.
